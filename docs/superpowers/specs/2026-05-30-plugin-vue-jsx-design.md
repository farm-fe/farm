# @farmfe/plugin-vue-jsx Design

## Summary

Port `swc-plugin-vue-jsx` to a native Farm Rust plugin (`@farmfe/plugin-vue-jsx`) that transforms Vue JSX/TSX at build time — converting directives (`v-model`, `v-show`, `v-html`, etc.) into Vue 3 runtime calls. The plugin lives as a standalone crate at `rust-plugins/vue-jsx/` and is distributed as `@farmfe/plugin-vue-jsx`.

## Motivation

Currently Farm users who use Vue JSX must configure `@vitejs/plugin-vue-jsx` via Farm's Vite plugin adapter. The swc-plugin-vue-jsx shipped as an SWC plugin (WASM bytecode), but Farm uses SWC as a native library via `farmfe_toolkit` — so we can integrate the visitor directly as a native Rust plugin, eliminating the WASM overhead and the Vite adapter dependency.

## Data Flow

1. **User configures** `import vueJsx from '@farmfe/plugin-vue-jsx'; plugins: [vueJsx({...})]`
2. Farm calls `FarmPluginVueJsx::new(config, options_json)` — deserializes Options
3. On each `.jsx`/`.tsx` module:
   - `transform` hook fires
   - Checks `param.module_type == ModuleType::Jsx || ModuleType::Tsx`
   - Parses source into SWC AST via `farmfe_toolkit::swc_ecma_parser::parse_file_as_module`
   - Constructs `VueJsxTransformVisitor` with options, unresolved_mark, and optional comments
   - Runs `module.visit_mut_with(&mut visitor)`
   - Codegens via `farmfe_toolkit::swc_ecma_codegen::to_code`
   - Returns transformed content with `module_type: Some(ModuleType::Tsx)` (or `Jsx`)

## Plugin Hook Surface

Only the `transform` hook. No `load`, `config`, or `update_modules` needed.

### transform

| Input | Type | |
|-------|------|---|
| `param.module_type` | `ModuleType::Jsx \| ModuleType::Tsx` | Only transforms JSX/TSX |
| `param.content` | `String` | Raw source |
| `param.resolved_path` | `&str` | File path |

| Output | Type | |
|--------|------|---|
| `content` | `String` | Transformed JS/TS |
| `module_type` | `Option<ModuleType>` | Preserved (Jsx→Jsx, Tsx→Tsx) |
| `source_map` | `Option<String>` | None initially |
| `ignore_previous_source_map` | `bool` | `true` (AST rewrite invalidates source maps) |

## Options (all 7 ported from swc-plugin-vue-jsx)

```ts
interface VueJsxPluginOptions {
  /** Convert `on` / `nativeOn` attrs to use `@vue/babel-helper-vue-transform-on` */
  transformOn?: boolean;           // default: false
  /** Inject PatchFlags for optimized VNode updates */
  optimize?: boolean;               // default: false
  /** Custom element detection patterns (regex strings) */
  customElementPatterns?: string[]; // default: []
  /** Merge attribute objects with `mergeProps` from Vue */
  mergeProps?: boolean;             // default: true
  /** Enable object slot detection via `_isSlot` helper */
  enableObjectSlots?: boolean;      // default: true
  /** Custom pragma (e.g. `h`), overrides `createVNode` */
  pragma?: string;                  // default: undefined (uses createVNode)
  /** Resolve TypeScript types for `defineComponent` props/emits */
  resolveType?: boolean;            // default: false
}
```

## Migration Change Log

### `HANDLER.with(...)` → `Result::Err`

The swc plugin uses `HANDLER.with(handler, handler.span_err(...))` for ~10 diagnostic sites:
- Missing v-text expr → TransformError
- Unresolvable type reference → TransformError with span info
- Bad v-models shape → TransformError

Each becomes `return Err(CompilationError::TransformError { resolved_path, msg })`.

### `swc_core::plugin::errors::HANDLER` → removed

Farm's plugin model doesn't have a thread-local error handler. All error sites return `Result::Err`.

### Comments infrastructure

`swc_plugin` provides `metadata.comments: Option<Comments>` from the SWC plugin runner. In Farm, SWC comments are available via the `swc_ecma_parser` parse options. We pass `Some(comments)` when available (parse phase preserves them), `None` otherwise. This matters for:
- `resolve_type`: uses `comments` to emit `/*#__PURE__*/` annotations on generated calls
- `pragma` search: reads `@jsx` pragma from leading comments

### Dependency mapping

| swc-plugin-vue-jsx | farmfe-plugin-vue-jsx |
|---|---|
| `swc_core` | `farmfe_core` + `farmfe_toolkit` + `farmfe_macro_plugin` |
| `bitflags` | `bitflags` (same crate) |
| `css_dataset` | `css_dataset` (same crate) |
| `fnv` | `fxhash` (Farm convention) |
| `indexmap` | `indexmap` (same crate) |
| `regex` | `farmfe_core::regex` |
| `serde` | `serde` (same crate) |

### `@vue/babel-helper-vue-transform-on`

When `transform_on: true`, the plugin emits `import _transformOn from '@vue/babel-helper-vue-transform-on'`. This is an external JS dependency — the user installs it in their project (matching the swc and Vite plugin behavior).

## Testing

Port all 43 test fixtures from swc-plugin-vue-jsx. Each fixture has `input.jsx` + `output.js` (+ optional `config.json`). Tests pattern:

```rust
#[test]
fn v_model_with_checkbox() {
  let plugin = make_plugin("{}");
  let input = include_str!("fixtures/v-model-with-checkbox/input.jsx");
  let expected = include_str!("fixtures/v-model-with-checkbox/output.js");
  let output = transform_jxs(&plugin, input, "test.tsx", &ModuleType::Tsx);
  assert_eq!(output, expected);
}
```

## Out of Scope

- **HMR**: Not needed — JSX transforms are idempotent, Farm's built-in HMR handles the resulting JS module
- **Vue SFC compilation**: Handled by existing `@farmfe/plugin-vue`
- **`@vue/babel-helper-vue-transform-on` bundling**: This is a user-land dependency, not shipped with the plugin

## Dependencies

- `farmfe_core` (re-exports swc_ecma_ast, swc_common, Config, Plugin trait, etc.)
- `farmfe_macro_plugin` (for `#[farm_plugin]`)
- `farmfe_toolkit` (re-exports swc_ecma_parser, swc_ecma_codegen, swc_ecma_visit, swc_atoms)
- `farmfe_toolkit_plugin_types` (SWC plugin proxy types — `swc_plugin::proxies::TransformPluginProgramMetadata` equivalent)
- `bitflags`, `css_dataset`, `indexmap`, `fxhash`, `regex`, `serde`
